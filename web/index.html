How CITYNEXUS works

CITYNEXUS is a 20×20 grid city. Every tick (which represents one in-game hour) five autonomous agents observe a partial slice of the world, decide what to do, and act. Their actions reshape the city; their effects feed back into observations.

The five agents

  • Delivery sees roads near pending package routes; runs BFS through visible roads avoiding blocked cells.
  • Traffic sees only intersections; broadcasts advisories when too many are congested.
  • Emergency sees discs around active accidents; dispatches the nearest idle ambulance to the highest-severity scene.
  • Police sees discs around incidents and accident hazards; dispatches cars and cordons severe protests.
  • Planner sees only aggregate metrics; raises per-role priorities when its summary view shows a load spike.

Adversarial scenarios

Each episode gets a scenario from the curriculum-driven generator. As difficulty rises, more shocks fire on a schedule:

  • Traffic spike — boosts density inside a disc.
  • Emergency cluster — drops 2–6 simultaneous accidents.
  • Blocked routes — places a batch of timed roadblocks.
  • Weather storm — forces stormy weather for N ticks (raises accident rate, reduces road throughput).

Process-aware rewards

Pure outcome rewards are too sparse to train on, so each agent gets a dense per-tick signal alongside terminal rewards:

  • Per-cell movement progress for any unit closing on its target.
  • Dispatch intent — credit for picking a high-severity target before arrival.
  • Planner anticipation — credit for elevating a priority that matches observed load.
  • Plus penalties for delays, collisions, and idle units when work is queued.

Persistent memory

High-risk zones (recurring accident or congestion hotspots), past failures, and successful strategies persist in localStorage. The Delivery and Planner agents query memory each tick to anticipate problems they've seen before. Toggle "Persistent memory" off to see the difference.

How rewards are gated

A 3-layer verifier inspects each tick:

  1. Programmatic — are the actions well-formed? (paths on roads, valid unit IDs, bounded coords)
  2. System state — did the claimed effect actually happen? (congestion really dropped, accidents really cleared)
  3. Semantic — were the priorities coherent? Did the highest-severity scenes get dispatched first?

Failures attribute to the responsible role, and that role's reward is zeroed for the tick (the others keep theirs).

Reading the live city

  • Zone color = land use; red overlay on roads = traffic density.
  • Pulsing pink X = accident, sized by severity.
  • Amber square = roadblock (timed).
  • Colored disks = mobile units; they move smoothly between ticks.
  • Purple hexagon = police-domain incident.
  • If weather effects are on: rain particles, storm tint + occasional flash.
Why this exists. CITYNEXUS was built for the OpenEnv hackathon as an end-to-end multi-agent training environment. The full Python implementation includes RL trajectory collection, baseline-vs-trained evaluation, an Unsloth/TRL-ready training pipeline, and OpenEnv compliance. This page is a faithful in-browser port of the simulation core — engine, agents, adversary, rewards, memory — so you can play with the dynamics without installing anything.
